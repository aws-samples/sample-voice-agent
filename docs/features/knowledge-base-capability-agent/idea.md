# Knowledge Base Capability Agent

## Problem Statement

The knowledge base search tool (`search_knowledge_base`) is currently embedded inside the voice agent process at `app/tools/builtin/knowledge_base_tool.py`. Its registration logic lives in `pipeline_ecs.py:_register_tools()`, its system prompt fragment is inline in `create_voice_pipeline()`, and its service dependency (`KnowledgeBaseService`) is tightly coupled to the voice agent's configuration loading (`ConfigService` / SSM). This means:

- Adding or modifying KB behavior requires redeploying the entire voice agent.
- The KB tool shares compute resources (CPU/memory) with the voice pipeline, STT, TTS, and all other tools.
- KB configuration (knowledge base ID, max results, confidence threshold) is baked into the voice agent's environment.
- Another team cannot own or iterate on the KB capability independently.

## Vision

Extract the knowledge base tool into an independent **Strands A2A capability agent** deployed as its own ECS Fargate service. The voice agent discovers it via CloudMap and routes tool calls to it over A2A within the VPC. The KB agent owns its own:

- Tool definition (via Strands `@tool` with explicit docstring)
- Agent Card (auto-generated by `A2AServer`)
- Bedrock Knowledge Base client and configuration
- Deployment lifecycle (independent CDK stack)
- IAM permissions (scoped to KB access only)

## Scope

### Tool to Extract

| Current Tool | Name | Category | Parameters |
|---|---|---|---|
| `knowledge_base_tool.py` | `search_knowledge_base` | `KNOWLEDGE_BASE` | `query` (string, required), `max_results` (integer, 1-5, optional) |

### Service Dependencies to Port

- **`KnowledgeBaseService`** (`app/services/knowledge_base_service.py`) -- Bedrock Knowledge Base retrieval client
- **Configuration:** `KB_KNOWLEDGE_BASE_ID`, `KB_RETRIEVAL_MAX_RESULTS`, `KB_MIN_CONFIDENCE_SCORE`, `AWS_REGION`

### What Stays in the Voice Agent

- The `FunctionCallFillerProcessor` -- covers latency during A2A calls
- The Pipecat tool handler registration -- handled by the A2A tool adapter from the dynamic capability registry
- The base system prompt -- no KB-specific prompt fragment needed (model-driven tool use)

## Technical Approach

### KB Agent Implementation

```python
from strands import Agent, tool
from strands.models import BedrockModel
from strands.multiagent.a2a import A2AServer

@tool
def search_knowledge_base(query: str, max_results: int = 3) -> dict:
    """Search the knowledge base for information about products, policies,
    procedures, or other documentation. Use this when the user asks questions
    that might be answered by company documentation, FAQs, or reference
    materials. Always cite the source when presenting information.

    Args:
        query: Natural language search query. Be specific for better results.
        max_results: Maximum results to return (1-5, default 3).
    """
    # Port from knowledge_base_tool.py:knowledge_base_executor
    # Uses KnowledgeBaseService.retrieve()

agent = Agent(
    name="Knowledge Base Agent",
    description="Searches enterprise knowledge base for product, policy, and procedure information",
    model=BedrockModel(model_id="us.anthropic.claude-haiku-4-5-20251001-v1:0"),
    tools=[search_knowledge_base],
)

a2a_server = A2AServer(agent=agent, host="0.0.0.0", port=8080)
a2a_server.serve()
```

### Infrastructure

- **CDK stack:** `infrastructure/src/stacks/kb-agent-stack.ts` using `CapabilityAgentConstruct`
- **CloudMap:** Auto-registers in `voice-agent-capabilities` namespace
- **IAM:** Bedrock KB query permissions (`bedrock:Retrieve`, `bedrock:RetrieveAndGenerate`)
- **Environment:** `KB_KNOWLEDGE_BASE_ID`, `KB_RETRIEVAL_MAX_RESULTS`, `KB_MIN_CONFIDENCE_SCORE` from SSM
- **Resources:** Minimal CPU/memory (256 CPU, 512 MB) -- KB queries are I/O-bound

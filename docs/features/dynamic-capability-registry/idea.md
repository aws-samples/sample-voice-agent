# Dynamic Capability Registry

## Problem Statement

The voice agent's capabilities (tools, RAG, observers, system prompt fragments) are currently hard-wired together in `pipeline_ecs.py`. Adding or changing a capability requires editing multiple files: importing the tool, adding conditional registration logic in `_register_tools()`, modifying the system prompt in `create_voice_pipeline()`, and updating `builtin/__init__.py`. Every capability is aware of every other capability at the pipeline assembly level, creating tight coupling that makes the system brittle and hard to extend.

This means:
- **Developers cannot easily add new capabilities** without understanding the entire pipeline assembly flow.
- **Capabilities cannot be composed per-call** -- the same static set is available for every conversation.
- **Each capability is not self-contained** -- RAG registration logic lives in the pipeline, not with the RAG code. Transfer tool prompt instructions are inline in the pipeline builder, not with the transfer module.
- **Swapping capabilities requires code changes** instead of configuration changes.
- **Different call phases cannot have different capabilities** -- e.g., a greeting phase might not need RAG, but a troubleshooting phase does.

## Vision

Each capability is an **independently deployed A2A-compliant service** built with the Strands Agents SDK. The voice agent discovers capabilities at runtime via AWS CloudMap and maps their A2A skills to Pipecat tools.

The core voice agent becomes a thin orchestration layer that:
1. Discovers available capability agents via CloudMap (background polling, 30-60s)
2. Fetches Agent Cards (auto-generated by Strands `A2AServer`)
3. Maps A2A skills to Pipecat tool definitions (Bedrock tool spec format)
4. Routes tool calls to remote agents via `A2AAgent.invoke_async()`

A new capability requires only deploying a new ECS service -- no voice agent code changes.

## Chosen Approach: Strands SDK + A2A + CloudMap

The initial local-plugin approach (below) was superseded by an A2A-based architecture. See `plan.md` for the full implementation plan. Key decisions:

- **Protocol:** A2A (JSON-RPC 2.0 over HTTP) via Strands SDK native support
- **Discovery:** AWS CloudMap HTTP namespace with ECS auto-registration
- **Agent framework:** Strands `Agent` + `A2AServer` (no custom base package)
- **Pipeline LLM:** Unchanged (`AWSBedrockLLMService`). Strands is only for remote agents.
- **Tool descriptions:** Model-driven from `@tool` docstrings. No custom prompt fragments.
- **Effort:** ~9 days (reduced from ~15 days by using Strands SDK)

## Original Proposed Solution (Superseded)

The following local-plugin approach was the initial design. It has been superseded by the A2A-based architecture in `plan.md`. Kept here for historical context.

### Capability as a First-Class Concept (Original)

A capability is a self-contained module that packages:
- **Tool definitions** (zero or more) -- the functions the LLM can call
- **System prompt fragment** -- instructions for the LLM on how/when to use this capability
- **Service dependencies** -- backing services the capability needs (KB client, CRM client, etc.)
- **Configuration schema** -- what environment variables / SSM parameters are required
- **Observers** -- any pipeline observers this capability needs
- **Lifecycle hooks** -- `on_register`, `on_call_start`, `on_call_end`, `on_phase_change`
- **Metadata** -- name, version, description, category, required capabilities, conflicting capabilities

### Capability Registry (Original)

A `CapabilityRegistry` discovers, validates, and manages capabilities:
- **Discovery**: Scan a `capabilities/` directory, load from config, or connect to external MCP servers
- **Dependency resolution**: Ensure required capabilities are present, detect conflicts
- **Configuration validation**: Verify all required config values are available before registration
- **Lifecycle management**: Initialize capabilities at call start, tear down at call end

### Per-Call Capability Composition (Deferred)

Different calls (or call phases) can activate different capability sets:
- A "greeting" phase might only need basic conversation
- A "troubleshooting" phase activates RAG + case management
- A "transfer" phase activates transfer tools + context packaging
- Capability sets can be driven by intent classification or explicit configuration

### MCP Integration (Deferred)

Capabilities could also be provided by external MCP servers, enabling:
- Remote capability discovery at runtime
- Third-party capability packages
- Shared capabilities across multiple agent deployments

## Technical Approach (Original -- Superseded by plan.md)

### Step 1: Define the Capability Interface (Original)

```python
class Capability(ABC):
    """Base class for all voice agent capabilities."""

    @property
    @abstractmethod
    def metadata(self) -> CapabilityMetadata:
        """Name, version, description, category, dependencies."""

    @abstractmethod
    def get_tools(self) -> list[ToolDefinition]:
        """Tool definitions this capability provides."""

    @abstractmethod
    def get_prompt_fragment(self) -> str | None:
        """System prompt instructions for this capability."""

    def get_observers(self) -> list[Any]:
        """Pipeline observers this capability needs."""
        return []

    def validate_config(self, config: dict) -> list[str]:
        """Return list of missing/invalid config keys."""
        return []

    async def on_register(self, context: PipelineContext) -> None:
        """Called when capability is registered with the pipeline."""

    async def on_call_start(self, call_context: CallContext) -> None:
        """Called at the start of each call."""

    async def on_call_end(self, call_context: CallContext) -> None:
        """Called at the end of each call."""
```

### Step 2: Refactor Existing Capabilities

Convert each existing tool group into a self-contained capability:

- **`KnowledgeBaseCapability`** -- bundles `knowledge_base_tool`, `KnowledgeBaseService`, KB system prompt instructions, and KB config validation
- **`TransferCapability`** -- bundles `transfer_tool`, SIP session management, transfer prompt instructions
- **`CRMCapability`** -- bundles customer lookup, case management, verification tools, CRM service, and CRM prompt instructions
- **`DiagnosticsCapability`** -- bundles test tools (random, echo, slow variants)

### Step 3: Capability Discovery and Registration

```python
class CapabilityRegistry:
    def discover(self, capabilities_dir: str) -> list[Capability]: ...
    def register(self, capability: Capability) -> None: ...
    def resolve_dependencies(self) -> DependencyGraph: ...
    def get_active_capabilities(self, call_config: CallConfig) -> list[Capability]: ...
    def compose_system_prompt(self, base_prompt: str, capabilities: list[Capability]) -> str: ...
    def compose_tools(self, capabilities: list[Capability]) -> list[ToolDefinition]: ...
```

### Step 4: Pipeline Integration

Replace `_register_tools()` and inline prompt assembly with:

```python
# In create_voice_pipeline():
registry = CapabilityRegistry()
registry.discover("app/capabilities/")  # auto-discover from directory
active = registry.get_active_capabilities(call_config)
system_prompt = registry.compose_system_prompt(BASE_PROMPT, active)
tools = registry.compose_tools(active)
observers = registry.compose_observers(active)
```

## Research Completed

Research items resolved during architecture discussion (2026-02-19):

1. **Pipecat plugin model**: Pipeline components are fixed at construction time. Tool handlers can be registered dynamically, but pipeline processors cannot be added/removed mid-session. Tool-level dynamism is sufficient for our needs.
2. **MCP integration with Pipecat**: Deferred. A2A is the chosen protocol for inter-agent communication.
3. **Strands Agents patterns**: Evaluated. Decision: use Strands SDK native A2A support (`Agent` + `A2AServer` for capability agents, `A2AAgent` for client communication). Auto-generates Agent Cards from `@tool` definitions. Eliminates need for custom base agent package.
4. **Phase-based capability switching**: Deferred to future iteration. Initial implementation uses per-session capability sets discovered via CloudMap.
5. **Performance implications**: A2A calls within VPC add ~30-50ms per tool call. Filler phrase system covers this latency. No per-call discovery overhead -- AgentRegistry polls in background every 30-60s.
6. **Prompt composition**: Decision: no per-capability prompt fragments. LLM uses model-driven tool selection from Bedrock tool specs alone (name, description, parameter schemas). Tool descriptions auto-generated from Strands `@tool` docstrings. This stays close to A2A standard and avoids custom extensions.
7. **Pipecat PR #2610 (StrandsAgentsProcessor)**: Evaluated and rejected for pipeline use. Would break filler phrases, observers, and token streaming. Strands is used only for remote capability agents, not as the pipeline LLM.

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Over-engineering for current scale | Start with minimal interface; iterate based on real needs |
| Breaking existing functionality during refactor | Implement behind feature flag; maintain backward compat |
| Pipecat limitations on dynamic pipeline changes | Research first; fall back to per-session capability sets |
| Complexity of dependency resolution | Keep it simple initially (flat list, no cycles) |
| Performance overhead at call start | Profile capability discovery; cache registry between calls |
